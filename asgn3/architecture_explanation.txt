The Backrooms: technical architecture explanation
=================================================

This file explains how the project is built from the first browser load to the
last frame of gameplay. The goal is to make the implementation easy to defend:
what data is stored, which functions change it, how WebGL draws it, and how the
game systems fit together.

1. What the browser loads first
-------------------------------

The page starts in index.html. It creates the Windows 95 style outer window,
the 800 by 600 game viewport, the WebGL canvas, the HUD, the title screen, the
VN dialogue box, the inventory panel, the warning sign popup, and the win/lose
screens. The title screen is inside #gameViewport, so in normal mode it appears
inside the fake desktop window instead of covering the whole browser. Fullscreen
mode hides the title bar and menu bar, removes the border, and stretches the
same viewport to the browser size.

The scripts are loaded in a fixed order because this project uses plain browser
globals rather than modules:

1. lib/cuon-matrix-cse160.js provides Matrix4 and Vector3.
2. webgl-utils.js provides the shader helper used by main.js.
3. shaders.js defines the GLSL source strings.
4. cube.js defines cube face constants and the shared vertex layout.
5. objLoader.js defines parseOBJ().
6. world.js defines the grid maps, level builders, and world rendering buffers.
7. camera.js defines the first-person Camera class.
8. entity_model.js provides baked Float32Array geometry for the creature.
9. animSystem.js loads anim.json and interpolates keyframes.
10. entity.js uploads, moves, pathfinds, and draws the creature.
11. gameState.js runs the story, inventory, item pickup, and VN UI.
12. main.js connects all of the above to WebGL, input, audio, and the frame loop.

The main entry point is main() in main.js. It looks up the canvas, creates the
WebGL context, compiles the shaders with initShaders(), stores all attribute and
uniform locations, enables depth testing, binds texture samplers to texture
units, loads textures, creates the Camera, initializes world buffers, loads
level 1, uploads the creature, loads animation JSON, starts loading OBJ meshes,
wires input/audio/UI, and finally starts requestAnimationFrame(tick).

2. The shared vertex format
---------------------------

Everything that is drawn as normal mesh geometry uses the same interleaved
vertex layout:

    x, y, z, u, v, nx, ny, nz

That is 8 floats per vertex and 32 bytes per vertex. The shader attribute
pointers always use:

- a_Position at byte offset 0
- a_TexCoord at byte offset 12
- a_Normal at byte offset 20

This is why cube faces, OBJ models, creature pieces, furniture, goop decals,
suburb geometry, and Backrooms chunks can all be drawn with the same shader
program and the same attribute setup.

cube.js defines FACE_FRONT, FACE_BACK, FACE_LEFT, FACE_RIGHT, FACE_TOP, and
FACE_BOTTOM. It stores six vertices for each face in FACE_VERTS and outward
normals in FACE_NORMALS. pushFace(arr, bx, by, bz, face) appends one cube face
at a grid-aligned block position. world.js uses pushFace() when it converts map
cells into vertex arrays. main.js also uses a single reusable cube buffer for
things that are easier to draw as one scaled cube, such as highlights, skybox,
and simple markers.

3. WebGL startup and the frame loop
-----------------------------------

main() reads all shader locations once. The important uniforms are:

- u_ModelMatrix, u_ViewMatrix, u_ProjectionMatrix for transforms.
- u_whichTexture for selecting a texture sampler.
- u_texColorWeight and u_baseColor for mixing solid color with texture.
- u_time for flicker and animation timing.
- u_lightPos[0] and u_numLights for point lights.
- u_flickerEnabled for the photosensitivity setting.
- u_isBackrooms for switching between steady suburb lighting and Backrooms
  flicker behavior.
- u_emissive for self-lit things such as ceiling lights, skybox, and TV glow.
- u_fogNear, u_fogFar, and u_fogColor for distance fog.

tick(timestamp) is called every frame. It clamps dt to at most 1/30 seconds so
a browser hitch does not move the player or entity a huge distance. While not
paused, tick() advances game time, the speedrun timer, the story state machine,
the round timer, input, entity AI, and entity audio. After updates it calls
renderScene(), updateHUD(), updateFps(), and schedules the next frame.

renderScene() draws in this order:

1. Clear color and depth buffers.
2. Upload camera view/projection matrices.
3. Upload light count, or zero lights during the short round reset blackout.
4. Draw the suburb skybox when level 1 is active.
5. Draw the grid world.
6. Draw level OBJ models such as fences, weeds, and sign.
7. Draw the garden highlight.
8. Draw simple level item cubes.
9. Draw goop decals.
10. Draw the exit door marker.
11. Draw furniture.
12. Draw the entity if the chase has started.

4. Camera, movement, and pointer input
--------------------------------------

camera.js stores a first-person camera as eye, at, up, yaw, pitch, viewMatrix,
and projectionMatrix. updateViewMatrix() converts yaw and pitch to a forward
direction, updates at, and calls Matrix4.setLookAt(). updateProjectionMatrix()
uses Matrix4.setPerspective() with a 70 degree field of view.

main.js handles input in setupInput() and processInput(). Keyboard state is
stored in g_keys. W and S move along the camera's flat forward vector. A and D
move along the flat right/left vector. Q and E rotate the camera from the
keyboard. Mouse look uses pointer lock: mousemove events add clamped deltas to
g_pendingMouseX and g_pendingMouseY, and processInput() drains only a bounded
amount per frame. That prevents camera snapping after browser hitches or after
pointer lock changes.

Movement is blocked by isBlockedAt(). It checks the four player-radius corners
against isWorldBlockedCell(), which looks at g_Map and g_CollisionMap. Furniture
also blocks movement by radius checks around each g_FurnitureSlots entry. The
actual movement uses tryMove(dx, dz), which tries the full movement first, then
tries X-only and Z-only movement. This gives wall sliding instead of stopping
the player completely when they scrape a corner.

The story can also block movement. canMoveToNarrative(nx, nz) prevents the
player from walking onto the porch before finishing the garden and keeps the
player near the sign after falling into the Backrooms. isNarrativeMovementLocked()
blocks all movement during fade screens, VN dialogue, and sign popup states.

5. Texture loading and fallbacks
--------------------------------

main.js uses six texture units:

- TEXTURE0 for Backrooms wall.png.
- TEXTURE1 for Backrooms floor and ceiling.
- TEXTURE2 for fluorescent light tiles.
- TEXTURE3 for the exit door texture.
- TEXTURE4 for transparent goop decals.
- TEXTURE5 for suburb textures and some OBJ model textures.

loadTexture() creates a WebGL texture, immediately uploads a procedural fallback
canvas, and then replaces it when the real image finishes loading. This means
the world can render even if an image file is missing or still loading.

loadBackroomsDoorTexture() combines the Minecraft oak door top and bottom image
files into one canvas before uploading it to TEXTURE3. The Backrooms wall cell
marked as DOOR_TYPE uses that sampler for the lower part of the door and wall
texture above it.

applyTexParams() changes its behavior based on performance mode. In normal mode,
power-of-two textures use LINEAR_MIPMAP_LINEAR for minification, LINEAR for
magnification, REPEAT wrapping when allowed, and gl.generateMipmap() after each
upload. Non-power-of-two textures use LINEAR and CLAMP_TO_EDGE because WebGL 1
cannot safely repeat or mipmap them. In performance mode, textures switch to
NEAREST min/mag filtering and skip mipmap use. setPerformanceMode() refreshes
the stored texture records so already-loaded textures change filtering when the
checkbox is toggled.

6. The shader and lighting model
--------------------------------

shaders.js has one vertex shader and one fragment shader.

The vertex shader multiplies each vertex by u_ModelMatrix, u_ViewMatrix, and
u_ProjectionMatrix. It passes texture coordinates, transformed normals, world
position, and camera-space distance to the fragment shader.

The fragment shader chooses a sampler based on u_whichTexture. It supports wall,
floor/ceiling, light tile, door, goop, and suburb texture samplers. Solid color
meshes use u_texColorWeight = 0.0. Textured meshes use u_texColorWeight = 1.0.
This is how the same shader can draw both textured walls and solid-colored OBJ
furniture or creature pieces.

Lighting is made from up to eight point lights stored in u_lightPos. pointLight()
uses distance attenuation and a diffuse normal term. fluorescent() changes light
intensity over time using a sine wave and a small hash-based random term. If
u_flickerEnabled is false, fluorescent() returns the steady base value. If
u_isBackrooms is false, lights stay steady for the suburb.

Fog is computed from v_Dist. The fragment color is mixed toward u_fogColor
between u_fogNear and u_fogFar. The suburb uses sky-colored fog. The Backrooms
uses yellow-brown fog. Goop textures use alpha, and transparent fragments are
discarded before lighting/fog output.

7. World data structures
------------------------

world.js stores the world as grid layers indexed by [x][z]. The important
layers are:

- g_Map: visible block height. 0 is open floor, positive values are stacked cube
  heights, and WALL_H is 4.
- g_MapType: material type for special wall cells, especially the exit door.
- g_CeilType: ceiling material, including fluorescent light tiles.
- g_CollisionMap: invisible collision, used for fences and story barriers.
- g_InteractiveMap: interaction code at a cell, such as weed, soil bag, sign,
  garden plot, hedge row, or watering can return point.
- g_SuburbFloorTex: per-cell floor texture for level 1.
- g_SuburbWallTex: per-cell wall/block texture for level 1.
- g_LevelItemSlots: small drawn items such as soil bag and watering can.
- g_LevelModelSlots: OBJ instances such as fence, weeds, and sign.
- g_FurnitureSlots: Backrooms furniture instances.

The level also exposes anchors such as SPAWN_X, SPAWN_Z, ENTITY_SPAWN_X,
ENTITY_SPAWN_Z, DOOR_CELL_X, DOOR_CELL_Z, DOOR_APPROACH_X, and DOOR_APPROACH_Z.
Other files read these anchors instead of hardcoding coordinates.

8. Level 1: hardcoded suburb map and garden tutorial
----------------------------------------------------

The assignment requires a hardcoded 2D JavaScript array. map_suburbs is a 32 by
32 row array in world.js. _copyRowsToXZ() converts it into g_Map[x][z]. Heights
in that map become cube stacks. The hand-authored map contains the house, yard,
porch area, and hedge cells.

initLevel1Map() resets all world layers for a 32 by 32 level. It assigns floor
textures by cell: road near the street, sidewalk, wood deck for the porch and
house floor, and grass elsewhere. It assigns housewall.png to normal walls and
grass.png to the hedge row. It fills g_CollisionMap for fence boundaries and
yard limits. It pushes fence OBJ instances into g_LevelModelSlots, places weed
OBJ instances in the garden, and marks those cells as INTERACT_WEED.

The suburb also creates the item and interaction setup for the tutorial:

- INTERACT_SOIL_BAG at cell (21, 6).
- INTERACT_WATER_CAN at cell (21, 9).
- INTERACT_GRASS_ROW for hedge cells at x = 8 and z = 7 through 9.
- g_LevelItemSlots for the visible soil bag and watering can cubes.
- g_LightPositions with four steady outdoor/house lights.
- The player spawn at (15.5, 1.5), facing the yard.

buildSuburbGeometry() turns the suburb grid into vertex buffers. It adds one
floor face for every cell. For each nonzero g_Map cell, it emits side faces only
where the neighbor is shorter, so hidden interior faces are not uploaded. It
also builds the house roof and the front door from cuboids. Visible item cubes
are added to texture batches too. The final output is g_suburbBatches, one WebGL
buffer per texture name, so renderSuburbWorld() can draw each texture batch with
TEXTURE5.

9. The OBJ parser
-----------------

objLoader.js contains parseOBJ(text). It is a small Wavefront OBJ parser built
for this project. It reads the OBJ file line by line and splits each trimmed
line into tokens.

- Lines beginning with v are pushed into positions.
- Lines beginning with vn are pushed into normals.
- Lines beginning with vt are pushed into uvs.
- Lines beginning with f are converted into triangles.

OBJ face tokens are 1-indexed, so parseOBJ subtracts 1 from each position, uv,
and normal index. It accepts v, v/vt, v/vt/vn, and v//vn style face tokens. If a
uv or normal is missing, it falls back to [0, 0] for uv and [0, 1, 0] for the
normal. Faces with more than three vertices are fan-triangulated: the parser
emits triangles [0, i, i + 1] for each i from 1 to n - 2. The result is a
Float32Array in the shared 8-float vertex layout.

The parser ignores .mtl material files because the project controls texture and
color through WebGL uniforms. That keeps the loader small and predictable for
assignment use.

10. Loading and placing OBJ models
----------------------------------

There are two OBJ loading paths in main.js.

The first path is for level models: fences, weeds, and the warning sign.
LEVEL_MODEL_DEFS maps a kind to an OBJ file and default color or texture. On
startup, loadLevelModelMeshes() fetches each OBJ, calls parseOBJ(), optionally
converts from Z-up to Y-up with _convertObjZUpToYUp(), computes a bounding box,
uploads the Float32Array into a WebGL buffer, and stores the mesh in
g_LevelModelMeshes. The bounding box is used to compute offsetX, offsetY, and
offsetZ so the model can be centered and placed on the ground.

world.js decides where these level models appear by pushing objects into
g_LevelModelSlots. For example, initLevel1Map() pushes multiple fence slots with
kind, x, z, yaw, length, height, and color. It also pushes weed slots for each
garden weed. setupBackroomsIntroSign() pushes the wooden sign into the
Backrooms near the spawn and marks the same cell as INTERACT_SIGN.

drawLevelModels() loops over g_LevelModelSlots. For each slot it looks up the
uploaded mesh, computes scale from slot width/height/length divided by the
mesh bounding box, translates to the slot's world position, rotates by yaw,
scales, applies centering offsets, and draws the buffer. If the slot or model
definition has a texture name, it binds that texture to TEXTURE5; otherwise it
uses solid color.

The second path is for furniture in the Backrooms. FURNITURE_DEFS maps chair,
chair1, chair2, chair3, chair4, sofa, tv, and tv_screen to OBJ files, colors,
target heights, and emissive flags. loadFurnitureMeshes() fetches and parses
each furniture OBJ, measures its height, computes a uniform scale to match the
target height, uploads the buffer, and stores centering offsets in
g_furnitureMeshes. initLevel2Map() fills g_FurnitureSlots with random room
positions, and drawFurniture() places each object with translate, yaw rotate,
scale, and offset. Furniture also participates in player and entity collision by
radius checks around each furniture slot.

11. Interaction targeting and item pickup/placement
---------------------------------------------------

Mouse input starts in onMouseDown(). The game only accepts clicks while pointer
lock is active. Left click is normal interact. Right click is used for the hedge
block breaking task. If the VN box or sign popup is open, the click closes that
UI first.

To find what the player clicked, getInteractionTargetCell() builds a 3D view ray
from camera yaw and pitch. It first checks actual object bounds with
getObjectInteractionTarget(). That function tests g_LevelModelSlots,
g_LevelItemSlots, and special interactive cells against rayAabbDistance(). This
is why a weed OBJ, soil bag cube, watering can cube, or sign can be selected by
looking at the visible object rather than only by looking at the cell center.

If no object AABB is hit, getInteractionTargetCell() checks the floor ray when
the player is looking down, then marches along the view ray in small steps from
0.25 to 4.2 units. The first relevant interactive grid cell wins. If nothing
interactive is found, it falls back to getTargetCell(), which returns the cell
directly in front of the player for door clicking.

gameState.js owns the meaning of each click. handleNarrativeInteract() routes
to handleSuburbInteract() in level 1 and opens the sign in the Backrooms. The
suburb interaction handler is phase-based:

- During TUTORIAL_WEEDS, INTERACT_WEED calls destroyGardenWeed(). That removes
  the weed model slot, changes the cell to a garden plot, and rebuilds the
  suburb geometry.
- During TUTORIAL_SOIL_PICKUP, INTERACT_SOIL_BAG calls removeInteractiveItem(),
  removes the soil bag slot, and puts SOIL BAG in the inventory with 6 uses.
- During TUTORIAL_SOIL_PLACE, clicking each INTERACT_GARDEN_PLOT calls
  placeGardenSoil(), changes that floor texture to dirt_unwatered.png, records
  the cell in g_gs.soilPlaced, and decrements the inventory count.
- During TUTORIAL_WATER_PICKUP, INTERACT_WATER_CAN removes the visible can and
  gives the player the watering can.
- During TUTORIAL_WATER_PLACE, clicking planted plots calls waterGardenPlot(),
  changes the texture to dirt_watered.png, records the cell in g_gs.waterPlaced,
  and empties the can after 6 uses.
- During TUTORIAL_RETURN_CAN, INTERACT_CAN_RETURN calls returnWateringCan(),
  puts the can back into g_LevelItemSlots, clears inventory, and starts the
  hedge task.
- During TUTORIAL_GRASS_ROW, right click calls breakHedgeBlock() on the middle
  hedge cell. That lowers g_Map[8][8] by one and adds a hedge block to the
  inventory. Left click calls placeHedgeBlock() on side hedge cells, which sets
  g_Map[x][z] to height 1. isHedgeRowComplete() checks that all three hedge
  cells are height 1, then the game advances to WALK_TO_HOUSE.

The important block-editing part is that block placement and removal are real
changes to g_Map. After each change, buildWorldGeometry() is called, so the WebGL
buffers match the new map state.

12. VN dialogue, popup text, and story UI
-----------------------------------------

The visual novel style UI is plain HTML and CSS layered over the canvas. The
main pieces are:

- #fadeOverlay and #fadeText for black fade screens and narration.
- #controlsToast for tutorial prompts.
- #vnBox, #vnAuthor, #vnText, and #vnContinue for dialogue.
- #inventorySlot, #invIcon, #invItemName, and #invUseCount for held items.
- #signPopup and #signCard for the Backrooms warning sign.

gameState.js updates these elements directly. showVN(text) puts text into
#vnText, shows #vnBox, and sets g_gs.vnOpen. hideVN() hides the box and marks
g_gs.vnJustDismissed so updateGameState() can advance to the next line. The E
key, Enter key, and clicks on the VN box all close it.

The sign popup works similarly, but it also releases pointer lock so the player
can click the CLOSE button. openBackroomsSign() shows #signPopup, sets the phase
to BACKROOMS_SIGN_OPEN, and suppresses the normal pointer-lock pause behavior.
closeBackroomsSign() hides the popup, plays a wail or fallback oscillator, then
starts the final VN lines before activating the entity.

The story itself is a finite-state machine stored in g_gs.phase. updateGameState()
advances timed parts such as the opening fade, falling into the Backrooms, and
the post-sign dialogue. setPhase() centralizes the side effects for each state:
objective text, toast text, body CSS classes, inventory clearing, and trap state.

13. Level loading and the fall into the Backrooms
-------------------------------------------------

loadLevel(levelID) is the central level swap function. For level 1 it calls
initLevel1Map(). For level 2 it calls initLevel2Map() and then
setupBackroomsIntroSign(). After the map is built, loadLevel() moves the camera
to the new spawn, resets yaw/pitch, resets the round timer, resets the entity
speed multiplier for the Backrooms, resets the entity path cache, rebuilds world
geometry, rebuilds goop if needed, uploads light positions, updates shader level
mode, updates fog/clear color, and sets body CSS classes.

The transition is handled by gameState.js. Once the player finishes the garden
and walks through the house door, the phase changes to FALLING. During that
phase, updateGameState() lowers camera.eye.y and tilts the camera downward. When
the fall timer finishes, it calls loadLevel(2), records the trap center, hides
the fade overlay, and enters BACKROOMS_TRAPPED. The player is then restricted to
a small area until they read the sign.

14. Procedural Backrooms generation
-----------------------------------

initLevel2Map() builds a new 96 by 96 Backrooms layout every time level 2 loads.
It starts by filling every cell with WALL_H, meaning the entire map is solid.
It uses a local linear congruential random number generator seeded from Date.now()
and Math.random(). The helper carve(x0, z0, x1, z1) turns rectangular areas into
open floor by setting g_Map[x][z] to 0.

The first step is anchor placement. The important anchor rooms are:

- Spawn room near (6, 6).
- Exit foyer near (90, 88).
- Entity spawn room near (48, 48).
- Extra landmark rooms in the northeast, southwest, and two mid-map positions.

These anchors make sure the main game loop always has a player start, an exit
area, and an entity start. The spawn and exit are far apart and not on the same
row or column, so the player cannot simply walk straight to the door.

After anchors, the generator adds random rooms. It creates small and medium rooms
until rooms.length reaches 55, with sizes from 4 by 4 to 6 by 6. It also creates
6 larger rooms from 8 by 8 to 10 by 10. Rooms may overlap; overlap creates open
merged spaces instead of a perfect maze.

The room graph is connected with carveCorridor(a, b). A corridor has width 2,
with a small chance of width 3. It forms an L shape: either horizontal then
vertical, or vertical then horizontal. The generator connects each room to the
next room, then adds 20 random extra connections, then connects every room to
its two nearest neighbor rooms. These extra paths create loops and reduce the
chance of isolated clusters.

After carving rooms and corridors, a flood-fill starts at the spawn anchor. Any
open cell not reached by that flood-fill is changed back to WALL_H. This keeps
the playable area connected.

The generator then adds obstacles inside rooms. pillarOK() only allows a pillar
where the pillar and a one-cell border around it are open, so the pillar stands
inside a room instead of touching another wall. Each non-spawn, non-exit room
can receive a 1 by 1 or 2 by 2 pillar. Some rooms also receive a partition wall
with a one-cell gap. These pillars and partitions are what make the Backrooms
feel broken up instead of like one giant open floor.

The exit door is placed after that. DOOR_CELL_X is 94 and DOOR_CELL_Z is 88. The
door cell is a wall cell with g_MapType set to DOOR_TYPE. A small foyer in front
of it is forced open from x 90 through 93 and z 87 through 89. The cell the
player must reach to click the door is doorReachX = DOOR_CELL_X - 1 and
doorReachZ = DOOR_CELL_Z.

The obstruction pass breaks up long straight sightlines. regionStep() chooses a
spacing value based on a hashed coordinate, so some rows/columns are more open
and others are denser. breakRow() and breakCol() find long runs of open cells
and try to plant walls along the run. tryBlock(bx, bz) rejects cells too close
to the spawn or door, places a wall temporarily, and runs bfsReach() from spawn
to the door approach. If the door becomes unreachable, the wall is rolled back.
That means the level can gain visual obstructions without breaking solvability.

After obstruction, a dead-end cleanup runs three passes. If an open cell has
only one open cardinal neighbor, the cleanup shuffles the four directions and
tries to carve one neighboring wall open. It again checks reachability before
keeping the carve. This removes many tiny dead ends produced by the obstruction
pass while preserving the main connected layout.

Lighting is placed after geometry. The generator marks light ceiling tiles in a
5-cell grid pattern with random skips. It then chooses up to eight actual point
light positions spread across a 3 by 3 partition of the map, because the shader
supports at most eight point lights. The spawn and door approach lights are
forced into g_LightPositions so both important areas are visible.

Finally, initLevel2Map() fills spawn/entity/door anchors and furniture. The
player starts at the spawn anchor, the entity starts at the center anchor, and
the door approach is near the southeast foyer. Big rooms receive furniture, and
ten extra random rooms may receive additional chairs. Furniture slots are later
drawn from OBJ buffers and also checked for collision.

15. Backrooms geometry and chunk culling
----------------------------------------

buildWorldGeometry() handles the Backrooms differently from the suburb. It
builds the 96 by 96 map in 8 by 8 chunks. For each chunk it creates four arrays,
one per material group:

- group 0: normal Backrooms walls
- group 1: floor and normal ceiling
- group 2: fluorescent light ceiling tiles
- group 3: exit door

appendBackroomsCellGeometry(groups, x, z) always emits a floor top face. If the
cell is open, it emits a ceiling bottom face and returns. If the cell is a wall,
it checks the four neighboring heights and emits only side faces that are visible
from a lower neighbor. Door cells use the door material for the lower door height
and normal wall material above that.

After a chunk's arrays are filled, each non-empty group is uploaded into its own
WebGL buffer. The chunk stores centerX, centerZ, radius, buffers, and counts.
renderWorldChunks() checks isWorldChunkVisible(chunk) before drawing. The cull
test uses distance from the camera, whether the chunk is behind the camera, and
a side-distance limit based on the camera forward vector. This is not exact
portal visibility, but it removes chunks behind and far outside the player's
view, which keeps the 96 by 96 level fast.

16. Goop decals and exit guidance
---------------------------------

Goop is stored separately from the world so it can be transparent and rebuilt
without changing the main chunk buffers. assignGoopTiles() uses a fixed seed,
0xF00DCAFE, so goop placement is deterministic for a given map pass. It walks
every open cell in g_Map.

For floor goop, a small chance creates a decal on that open cell. Most selected
floor decals are directional arrows. getFloorArrowTex(fx, fz, wrong) computes
the vector from the floor cell to the exit door, uses atan2() to choose one of
eight compass sectors, and returns the matching arrow texture. GOOP_WRONG_CHANCE
is 0, so arrows are not deliberately misleading. Non-arrow floor decals are
chosen from GOOP_FLOOR_DECORS.

For wall goop, the code checks four neighbor directions from each open cell. If
the neighbor is a wall, that wall face is visible from the open cell, so it can
receive a decal. Some wall decals are decorative. Directional wall decals choose
left or right based on the tangent direction along the wall face and the dot
product toward the exit. That makes wall arrows generally point along the wall
toward the door.

addExitBreadcrumbs() always places several arrows near the door approach, even
if random goop placement did not happen there. This makes the exit readable at
the end of the chase.

buildGoopGeometry() groups goop faces by the same 8 by 8 chunk grid as the world
and by texture name. It uses pushFace() to create floor or wall faces, uploads
each texture batch into a buffer, and stores g_goopChunks. drawGoopWorld()
enables blending, disables depth writes, applies polygon offset to avoid
z-fighting with the floor/wall underneath, culls chunks with isWorldChunkVisible(),
binds each goop texture on TEXTURE4, and draws the batches.

17. Entity model, animation, and drawing
----------------------------------------

The creature geometry is stored in entity_model.js as baked Float32Array data.
It has separate arrays for body, head, left arm, right arm, left leg, and right
leg, plus a merged fallback mesh. initEntityModel() in entity.js uploads every
available part into its own WebGL buffer. If the six-part version is not
available, drawEntity() can still draw the fallback buffer.

animSystem.js loads anim.json with XMLHttpRequest. The JSON contains a duration
and per-node keyframes. _findBracket() finds the two keyframes around the
current animation time. getInterpolatedTransformInto() wraps time by duration,
linearly interpolates translation and Euler rotation, and builds a local matrix
around the node pivot. The matrix order is translate, move to pivot, rotate Z,
rotate Y, rotate X, move back from pivot.

_drawEntityHierarchical(seconds) builds the creature as a hierarchy. First it
builds the world placement matrix from entity x/z position, yaw, scale, and OBJ
centering offsets. Then it gets the animated body local matrix, multiplies it by
the world matrix, and draws the body. For each child part, it gets that child's
animated local matrix, multiplies it by the body matrix, and draws the part.
That makes the head, arms, and legs inherit body movement while still animating
from their own pivots.

The model scale uses the baked OBJ bounding height. E_SCALE is 2.4 divided by
E_OBJ_H, so the creature appears about 2.4 game units tall.

18. Entity pathfinding and chase behavior
-----------------------------------------

The entity does not directly walk through walls. entity.js creates a reusable
pathfinding cache with _ensurePathBuffers(). It allocates typed arrays sized to
MAP_SIZE * MAP_SIZE:

- g_pathStepX and g_pathStepZ store the next step direction for each cell.
- g_pathSeen stores a stamp value so the code does not need to clear the whole
  array every search.
- g_pathQueueX and g_pathQueueZ store the BFS queue.

_recomputePath() runs BFS from the player's current grid cell, not from the
entity. This builds a flow field over the reachable map. When BFS visits a
neighbor cell (nx, nz) from current cell (cx, cz), it stores:

    g_pathStepX[idx] = cx - nx
    g_pathStepZ[idx] = cz - nz

That stored direction points from the neighbor back toward the cell closer to
the player. Later, when the entity is standing in that neighbor cell, it can
look up one step and move toward the player without storing a whole path list.

The BFS first checks four cardinal neighbors. It also checks diagonal neighbors,
but it rejects a diagonal if either adjacent cardinal side is blocked. That
prevents corner cutting through wall corners. The BFS is recomputed when the
timer reaches zero, when there is no path yet, or when the player changes grid
cell. Otherwise it reuses the current flow field for about 0.35 seconds.

updateEntity(dt) chooses the actual target. If _hasLineOfSightToPlayer() is
true, it targets the player's exact eye x/z position. If there is no line of
sight, it looks up the entity's current grid cell in the BFS arrays and targets
the center of the next step cell. If no step exists, it falls back to the player
position.

The entity then computes a normalized direction, smoothly rotates yaw toward the
movement direction, and moves by speed * dt. It uses _entityBlockedAt() to test
walls and furniture. Like the player, it tries full movement first, then X-only
and Z-only movement, so it can slide along corners. If blocked, it forces an
early path recalculation.

Line of sight is a grid ray. _hasLineOfSightToPlayer() ignores targets farther
than 24 units, samples from entity to player every 0.5 units, floors each sample
to a grid cell, and returns false if any g_Map cell is a wall. If line of sight
is true and the player is within ENTITY_BURST_RANGE, the entity uses
ENTITY_BURST_MULT. Every time the round timer wraps, g_entitySpeedMult also
increases, making the chase harder the longer the player takes.

19. Audio and game pressure
---------------------------

Audio is initialized only after the player presses ENTER on the title screen,
which satisfies browser autoplay rules. initAudio() creates an AudioContext.
initAudioFiles() starts the looping ambience element, creates the chase loop
element, and loads footstep and wail WAV files into Web Audio buffers.

The player footstep sound is driven by movement in processInput(). The entity
footstep sound is driven by g_entity.vx and g_entity.vz in updateEntityAudio().
The entity step cadence is faster during chase mode. Volume fades with distance.
The wail uses a cooldown so it only fires occasionally when the entity has line
of sight. The chase loop fades in while the entity sees the player and fades out
when line of sight is lost.

The round timer only counts down once the Backrooms chase is active. If it hits
zero, the round number increases, the timer resets to 120 seconds, lights cut
out briefly, and g_entitySpeedMult increases. Winning happens when the player
left-clicks the exit door cell. Losing happens when getEntityDist() falls below
1.5.

20. Performance choices
-----------------------

The project uses several performance choices that matter for a 96 by 96 world:

- Static world geometry is uploaded into WebGL buffers instead of rebuilt every
  frame.
- Hidden cube faces are skipped before upload.
- Backrooms geometry is split into 8 by 8 chunks.
- World and goop chunks share the same camera-distance culling logic.
- Furniture and simple props are culled by distance and forward direction.
- Entity pathfinding reuses typed arrays instead of allocating arrays every BFS.
- Entity drawing reuses Matrix4 objects.
- HUD updates cache DOM element references and avoid repeated text/style writes
  when values have not changed.
- Normal mode uses bilinear filtering and mipmaps on power-of-two textures;
  performance mode switches textures to NEAREST and avoids mipmap sampling.
- Mouse deltas are capped per event, capped while pending, and capped per frame.

Performance mode sets the canvas backing resolution to 40 percent of the CSS
display size, tightens fog distances, and switches texture sampling to cheaper
nearest-neighbor filtering. It does not use different world culling or hide
different geometry. Normal mode and performance mode share the same optimized
draw paths.

21. Window frame and fullscreen behavior
----------------------------------------

The game is normally bounded inside a fake Windows 95 style window. The outer
#container has gray bevel borders, a blue title bar, a menu bar, and an inset
#gameViewport panel. The title bar says The Backrooms. The splash screen,
warning, controls, and credits are all inside #gameViewport.

setupFullscreen() implements tab fullscreen with CSS instead of native browser
fullscreen. setGameFullscreen(true) toggles body.game-fullscreen, hides the
title bar and menu bar, removes the frame border and shadow, stretches the
viewport and canvas to window.innerWidth and window.innerHeight, updates the
WebGL viewport, and refreshes the camera projection matrix. Escape remains the
game pause key because the browser is not using native fullscreen mode.

22. Assignment requirements covered
-----------------------------------

The required 32 by 32 map is the hardcoded suburb map in map_suburbs. The
Backrooms is larger at 96 by 96. The world is drawn from cube faces with texture
coordinates. The camera is first-person and perspective. Movement supports WASD,
Q/E turning, and mouse look. Multiple textures are loaded and selected in the
shader. Blocks can be removed and placed during the hedge task by changing
g_Map. The entity is the moving creature. The OBJ loader is implemented in
objLoader.js and used for props, furniture, weeds, fences, and the sign. The
story layer adds the garden tutorial, fall, warning sign, chase, timer, win, and
lose states. The scene also includes lighting, fog, audio, fullscreen UI,
procedural generation, and an animated enemy for the extra polish requirement.
